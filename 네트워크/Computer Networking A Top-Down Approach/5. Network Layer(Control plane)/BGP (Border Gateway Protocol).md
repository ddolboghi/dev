- 인터넷에서 모든 AS(자율 시스템)은 동일한 AS간 라우팅 프로토콜(BGP)을 실행한다.
- BGP는 거리 벡터 라우팅의 맥락에서 탈중앙화 및 비동기적 프로토콜이다.
- BGP는 복잡한 프로토콜이지만, 인터넷을 깊이 이해하려면 그 기초와 동작 방식에 익숙해져야 한다.

# BGP의 역할
BGP에서 목적지는 `138.16.68/22`와 같은 형태를 가질 수 있으며, 이는 1,024개의 IP 주소를 포함한다. 따라서 라우터의 포워딩 테이블에는 `(x, I)` 형태의 항목이 포함되며, 여기서 `x`는 접두사(`138.16.68/22`)이고 `I`는 라우터 인터페이스 번호이다.

BGP는 라우터가 수행해야 하는 두 가지 중요한 작업을 위한 수단을 제공한다.
## BGP가 라우터에게 제공하는 것
### 이웃 AS로부터 접두사(prefix) 도달 가능성 정보를 얻을 수 있다
특히 BGP는 각 서브넷이 자신의 존재를 나머지 인터넷에 알릴 수 있도록 허용한다. BGP가 없다면 각 서브넷은 고립된 섬이 되어 인터넷의 나머지 부분과 통신할 수 없다.
### 접두사로 가는 최상의 경로를 결정한다
라우터는 특정 접두사에 대해 두 개 이상의 다른 경로 정보를 학습할 수 있다.
최상의 경로를 결정하기 위해 라우터는 이웃 라우터로부터 얻은 **접두사 도달 가능성 정보**와 **정책**을 기반으로, BGP 경로 선택 절차(route-selection procedure)를 실행한다.

## BGP 경로 전파
AS3에 있는 접두사 `X`에 대한 도달 가능성 정보가 다른 AS들의 모든 라우터에 전파되는 과정:
1. AS3는 AS2에게 "AS3 `X`"라는 BGP 메시지를 보내 `X`가 AS3에 있다고 알린다.
2. AS2는 AS1에게 "AS2 AS3 `X`"라는 BGP 메시지를 보내 `X`에 도달하려면 AS2를 거쳐 AS3로 가야 함을 알린다.
## eBGP와 iBGP
실제로는 AS가 아닌 라우터들이 서로 메시지를 주고받는다.
- **BGP 연결 (BGP connection)**: 라우터들은 **반영구적인 TCP 연결(포트 179)** 을 통해 라우팅 정보를 교환한다. 이 TCP 연결과 그 위에서 전송되는 모든 BGP 메시지를 BGP 연결이라고 한다.
- **eBGP (external BGP)**: 서로 다른 두 AS에 걸친 BGP 연결이다. 일반적으로 서로 다른 AS의 게이트웨이 라우터를 직접 연결하는 각 링크마다 하나의 eBGP 연결이 있다.
- **iBGP (internal BGP)**: 동일한 AS 내의 라우터 간 BGP 세션이다. iBGP 연결은 물리적 링크와 항상 일치하지는 않는다.

도달 가능성 정보는 eBGP와 iBGP 세션을 모두 사용하여 전파된다.
![[eBGP_iBGP.png]]
위 그림에서 접두사 `x`에 대한 정보는 다음과 같이 전파된다.
1. 게이트웨이 라우터 3a가 2c에게 eBGP 메시지 "AS3 `x`"를 보낸다.
2. 게이트웨이 라우터 2c는 2a를 포함한 AS2 내의 다른 모든 라우터에게 iBGP 메시지 "AS3 `x`"를 보낸다.
3. 게이트웨이 라우터 2a는 1c에게 eBGP 메시지 "AS2 AS3 `x`"를 보낸다.
4. 게이트웨이 라우터 1c는 iBGP를 사용하여 AS1 내의 모든 라우터에게 "AS2 AS3 `x`" 메시지를 보낸다.

이 과정이 끝나면 AS1과 AS2의 각 라우터는 `x`의 존재와 `x`로 가는 AS 경로를 알게 된다.
실제 네트워크에서 라우터는 `x`로 가는 여러 경로 정보를 받을 수 있다.
# 최적의 경로 결정하기
## BGP 경로 속성
라우터가 BGP 연결을 통해 접두사 정보를 알릴 때, 여러 **BGP 속성**도 함께 알린다. 접두사와 그 속성을 합쳐 **경로(route)** 라고 부른다. 중요한 속성 두 가지는 `AS-PATH`와 `NEXT-HOP`이다.

- **AS-PATH**: 이 속성은 접두사 정보가 거쳐온 AS들의 목록이다.
	- 접두사가 특정 AS로 전달될 때마다 해당 AS는 자신의 ASN(자율 시스템 번호)을 `AS-PATH` 목록에 추가한다. 
	- 이 속성은 라우팅 루프를 방지에 사용된다. 라우터가 자신의 AS가 경로 목록에 포함된 것을 보면, 그 정보를 거부한다.
- **NEXT-HOP**: 외부 AS로 가는 첫 관문 라우터의 IP 주소다.
	- AS 간 라우팅과 AS 내부 라우팅을 연결한다. 
 
위 그림에서 AS1의 라우터들은 접두사 `x`에 대해 두 개의 BGP 경로를 학습한다. 각 경로는 `(NEXT-HOP, AS-PATH, 목적지 접두사)`로 구성된다.
## Hot Potato Routing
- **Hot Potato 라우팅**은 자신의 AS 내에서 `NEXT-HOP` 라우터까지의 비용이 가장 적은 경로를 선택하는 라우팅 알고리즘이다. 
- 라우터는 자신의 AS 내부 라우팅 정보(e.g. OSPF)를 참조하여 각 `NEXT-HOP` 라우터까지의 최소 비용 경로를 찾고, 그중 가장 작은 비용을 가진 경로를 선택한다.
- 자신의 AS 내에서의 비용을 줄이려는 이기적인 알고리즘으로, AS 외부의 경로 비용은 무시한다. 이 때문에 같은 AS 내의 다른 라우터들이 동일한 접두사에 대해 서로 다른 AS 경로를 선택할 수도 있다.

## 실제 경로 선택 알고리즘
라우터가 특정 접두사에 대해 두 개 이상의 경로를 학습하면, 하나의 경로만 남을 때까지 다음 규칙들을 순서대로 적용한다.

1. **로컬 선호도 (Local Preference)**: 각 경로에는 로컬 선호도 값이 할당된다. 이 값은 AS의 네트워크 관리자가 정책에 따라 설정한다. 가장 높은 로컬 선호도 값을 가진 경로가 선택된다.
2. **최단 AS-PATH**: 남아있는 경로들 중에서 `AS-PATH`가 가장 짧은 경로가 선택된다.
3. **Hot Potato 라우팅**: 여전히 여러 경로가 남아있다면, `NEXT-HOP` 라우터가 가장 가까운 경로를 선택한다.
4. **BGP 식별자**: 그래도 여러 경로가 남으면, 라우터는 BGP 식별자를 사용하여 경로를 선택한다.

이 알고리즘에서 `AS-PATH` 규칙이 Hot Potato 라우팅 규칙보다 먼저 적용되기 때문에, 먼저 AS 경로가 짧은 경로를 찾아 종단 간 지연을 줄이려고 시도하므로 BGP는 단순히 이기적인 알고리즘이 아니다.
# IP-Anycast
IP-Anycast는 **하나의 동일한 IP 주소를 전 세계 여러 위치에 있는 여러 서버가 함께 사용**하는 네트워크 기술이다.
BGP는 인터넷의 AS 간 라우팅 프로토콜일 뿐만 아니라, **IP-Anycast**를 구현하는 데에도 사용된다.
IP-Anycast는 주로 DNS 시스템에 사용된다.
## BGP를 이용한 IP-Anycast 구현
![[IP-anycast_CDN.png]]
1. **동일 IP 주소 할당**: CDN 회사는 지리적으로 분산된 각 서버에 **동일한 IP 주소**를 할당한다.
2. **BGP 경로 광고**: 각 서버는 표준 BGP를 사용하여 이 동일한 IP 주소에 대한 경로를 광고한다.
3. **경로 선택**: BGP 라우터가 이 IP 주소에 대한 여러 경로 광고를 수신하면, 이들을 서로 다른 물리적 위치가 아닌 **동일한 물리적 위치로 가는 다른 경로로 취급**한다. 각 라우터는 로컬에서 BGP 경로 선택 알고리즘을 사용하여 해당 IP 주소로 가는 최상의 경로(e.g. AS-hop 카운트 기준 가장 가까운 경로)를 선택하고 자신의 라우팅 테이블에 설정한다.
4. **라우팅**: 클라이언트가 해당 IP 주소로 요청을 보내면, 인터넷 라우터들은 BGP 경로 선택 알고리즘에 의해 결정된 **가장 가까운** 서버로 요청 패킷을 포워딩한다.
## DNS 시스템에서 IP-Anycast 사용
실제 CDN은 BGP 라우팅 변경으로 인해 동일한 TCP 연결의 패킷들이 서로 다른 서버 인스턴스에 도착할 수 있기 때문에 일반적으로 IP-anycast를 사용하지 않는다.

IP-anycast는 DNS 시스템에서 DNS 쿼리를 가장 가까운 루트 DNS 서버로 보내기 위해 광범위하게 사용된다. 13개의 루트 DNS 서버 IP 주소 각각에 대해 전 세계에 흩어져 있는 수많은 DNS 루트 서버들이 존재하며, IP-anycast는 이들 중 가장 가까운 서버로 DNS 쿼리를 라우팅한다.
# 라우팅 정책
라우터가 경로를 선택할 때, **AS 라우팅 정책은 기술적인 경로 선택 기준보다 우선한다**.
로컬 AS의 정책에 의해 결정되는 `local preference` 속성이 이를 구현하는 데 사용된다.
## Customer/Provider 관계 구현
![[BGP_정책_시나리오.png]]
위 그림에서, X는 B와 C라는 두 제공자에게 연결된 **multi-homed access ISP**이다. 
X는 B와 C 사이의 트래픽을 중계하는 통과 네트워크(transit network)가 되어서는 안 된다. 
**액세스 ISP가 자신에게 도달하는 경로만 외부로 광고**하는 정책을 사용하면 고객/제공자 관계를 구현할 수 있다.
- X는 이웃인 B와 C에게 자신 외의 다른 목적지로 가는 경로가 없다고 광고한다. 즉, X가 `XCY`를 통해 Y 네트워크에 도달하는 경로를 알고 있더라도, 이 경로를 B에게 광고하지 않는다. 
- B는 X가 Y로 가는 경로를 가지고 있다는 사실을 모르기 때문에 Y로 가는 트래픽을 X를 통해 전달하려고 시도하지 않는다.
## Provider 네트워크 간의 정책
B가 A로부터 W로 가는 경로 `AW`를 학습했다고 가정하자.
B는 이 경로를 자신의 고객인 X에게는 `BAW`로 광고하여 X가 W로 갈 수 있음을 알린다.

하지만 B는 이 경로를 또 다른 제공자 네트워크인 C에게 광고해야 할까?
만약 광고한다면 C는 `BAW`를 통해 W로 트래픽을 보낼 수 있게 된다. B는 A와 C 사이의 통과 트래픽을 운반하고 싶지 않을 수 있다.

백본 ISP들은 일반적으로 자신의 Customer와 관련된 트래픽만 처리하며, **다른 백본 ISP들 사이의 통과 트래픽은 전달하지 않으려는 정책**을 사용한다. 그렇지 않으면 트래픽이 ISP 네트워크를 무료로 이용하는 셈이 된다.
이러한 관계는 **피어링 협정**을 통해 결정된다. 개별적인 피어링 협정은 일반적으로 ISP들 간에 협상되며, 종종 기밀로 유지된다.
# 통합 정리
IP 주소 지정, DNS, BGP를 포함하여 지금까지 본 많은 프로토콜과 개념들을 통합하여 설명한다.
## 인터넷에 연결되기 위한 단계
새로운 회사가 자신의 웹 서버, 메일 서버, DNS 서버를 전 세계에서 접근 가능하게 하려면 다음과 같은 단계를 거쳐야 한다.

1. **ISP와 연결 및 IP 주소 블록 할당받기**: 먼저 로컬 ISP와 계약하여 물리적 연결(e.g. DSL, 임대 회선)을 확보해야 한다. ISP는 회사에 IP 주소 범위(e.g. `/24` 주소 범위)를 제공한다. 할당받은 IP 주소는 웹 서버, 메일 서버, DNS 서버, 게이트웨이 라우터 등 회사의 네트워크 장치에 할당된다.

2. **도메인 이름 등록 및 DNS 서버 정보 제공**: 인터넷 등록기관과 계약하여 회사 도메인 이름을 얻어야 한다. 이때 회사의 DNS 서버 IP 주소를 등록기관에 제공해야 한다. 등록기관은 이 정보를 `.com`과 같은 최상위 도메인(TLD) 서버에 등록한다. 이 단계가 완료되면, 전 세계 누구나 회사의 도메인 이름을 통해 회사의 DNS 서버 IP 주소를 얻을 수 있게 된다.

3. **자체 DNS 서버에 레코드 생성**: 사람들이 회사의 웹 서버 IP 주소를 찾을 수 있도록, 회사의 DNS 서버에 웹 서버의 호스트 이름(예: `www.xanadu.com`)을 IP 주소에 매핑하는 항목을 추가한다. 메일 서버와 같은 다른 공개 서버에 대해서도 유사한 항목이 필요하다.

4. **BGP를 통한 접두사 광고**: 외부에서 회사의 서버로 가는 IP 데이터그램이 올바르게 라우팅되려면, 인터넷의 모든 라우터가 회사의 `/24` 접두사의 존재를 알아야 한다. 이는 BGP를 통해 이루어진다. 회사의 로컬 ISP는 BGP를 사용하여 회사의 접두사를 자신이 연결된 다른 ISP들에게 광고한다. 이 ISP들은 다시 BGP를 통해 이 광고를 전파한다. 결국, 모든 인터넷 라우터들은 회사의 접두사를 학습하게 되어 회사의 웹 및 메일 서버로 향하는 데이터그램을 올바르게 포워딩할 수 있게 된다.
